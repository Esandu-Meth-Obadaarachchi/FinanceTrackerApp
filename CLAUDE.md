# FinTrack — Claude guide

A Flutter app (web + Android) backed by Firebase (Auth + Firestore). The UI
is a faithful port of `Financial Tracker _Standalone_.html`. Use this file
as your map of conventions and gotchas before editing.

## Stack

- **Flutter** 3.32+ / Dart 3.8 — material widgets, no flutter_localizations.
- **Firebase** — `firebase_core`, `firebase_auth`, `cloud_firestore`.
- **Google Sign-In** — `google_sign_in` (mobile native picker, web popup).
- **State** — `provider` (ChangeNotifier). No Riverpod, no Bloc.
- **Fonts** — `google_fonts` (DM Sans for text, DM Mono for figures).
- **Local prefs** — `shared_preferences` (theme only).
- **CSV export** — `share_plus` (web download + native share).

`pubspec.yaml` is the source of truth — keep deps pinned with caret.

## Critical patterns

### Modals must re-expose AppState
`AppState` is provided in [main.dart `AuthGate`](lib/main.dart) — **below** the
root navigator. `showModalBottomSheet` and `showDialog` insert the route
**above** that provider, so a naive `context.read<AppState>()` inside a sheet
throws `ProviderNotFoundException` and the app crashes.

Every sheet entry point goes through `_showWithState(context, sheet)` in
[`screens/modals/sheets.dart`](lib/screens/modals/sheets.dart), which
re-wraps the builder in `ChangeNotifierProvider<AppState>.value`. Add new
sheets the same way. `ThemeController` is provided above MaterialApp so it
is reachable from sheets unchanged.

### Account balance is computed, never stored
`Account.openingBalance` is the only persisted figure. The current balance
is `AppState.balanceOf(account)` — opening + the effect of every relevant
transaction (income received +, expense -, transfer - source / + dest),
**minus any pending lent loan** drawn from that account. Don't add a mutable
`balance` field; add a transaction (or loan) and let the reducer math
reflect it.

### Lent money leaves the account until repaid
A `Loan` with `loanType == 'lent'` and `status == 'pending'` is money that
has physically left its `accountId`, so `balanceOf()` subtracts it (and
`totalBalance` drops with it). Marking the loan repaid clears the pending
state, which restores the balance automatically — there is no separate
repayment transaction. Borrowed loans are intentionally **not** reflected in
balances. `AppState.totalLent` sums the outstanding (pending) lent amount;
the dashboard surfaces it alongside pending income in the "To Receive" card.

### Recurring transactions are materialized client-side
A `recurring/{ruleId}` rule is a template; the actual entries are normal
`transactions` documents tagged with `recurringId`. `AppState._materializeRecurring()`
(no Cloud Functions — free tier) runs once accounts + transactions + recurring
streams are all ready and again whenever they change. For each `active` rule it
generates one transaction per month from `startMonth` (or the month after
`lastGeneratedMonth`) up to the current calendar month, then bumps
`lastGeneratedMonth`. That high-water mark is the idempotency key: it survives
across sessions/devices and means a **deleted** generated entry is **not**
recreated. A `_materializing` lock + in-session `_generated` set guard against
the re-entrancy caused by our own writes re-triggering the stream. Editing a
rule only affects future months; per-month edits are just edits to the
generated transaction. Reach the manager via `openRecurringManager(context)`
(re-provides `AppState`, like sheets do) from the Transactions screen.

### Per-user Firestore layout
Each user owns a `users/{uid}/{accounts,transactions,loans}` subtree; rules
restrict access to the owner. Full field-by-field schema, ordering and deploy
command live in the **Database architecture** section below.

### Phone-frame layout on wide screens
`main.dart`'s `MaterialApp.builder` wraps every route in
`Center > ConstrainedBox(maxWidth: 480)` on a `Brand.backdrop` background.
Keep new top-level routes happy with that: don't assume full viewport width.

### Theme & colour helpers
Read colours with `context.watch<ThemeController>().colors` (a `Palette`).
Constants that don't change with theme (income green, expense red, gradients)
live on `Brand` in [palette.dart](lib/theme/palette.dart). Don't hardcode
hex inside screens unless it matches one of those.

### Google sign-in
[`AuthService.signInWithGoogle()`](lib/services/auth_service.dart) uses
`FirebaseAuth.signInWithPopup` on web (`kIsWeb`) and `google_sign_in`'s
native picker on mobile, then exchanges the Google credential with
FirebaseAuth. `signOut()` also calls `GoogleSignIn.signOut()` on mobile so
the next sign-in re-prompts for the account.

Android needs the debug/release keystore **SHA-1** registered on the
Firebase Android app, otherwise the Google picker fails with
`DEVELOPER_ERROR` / code 10. After adding a fingerprint, re-run
`flutterfire configure --project=fintrack-05220041 --platforms=android,web --yes`
to pull a fresh `google-services.json` (it'll contain an `oauth_client`
with `client_type: 1` once the SHA-1 is registered).

### Brand logo
The PNG at [`assets/logo.png`](assets/logo.png) is rendered via
`Image.asset` on the auth screen and also copied (manually) to
`web/favicon.png`, `web/icons/Icon-*.png`, and the
`android/app/src/main/res/mipmap-*/ic_launcher.png` set. When the logo
changes, update the source file and re-copy to those locations.

### Modern Color API
Flutter 3.27+ deprecated `Color.value` and `withOpacity`. Use:
- `color.toARGB32()` (see [utils/color_x.dart](lib/utils/color_x.dart))
- `color.withValues(alpha: 0.x)`

## Database architecture

Cloud Firestore (region `asia-south1`), one document tree per user. There is
no server backend — the Flutter client talks to Firestore directly and
`firestore.rules` is the only access boundary.

### Collection tree
```
users/{uid}                       — implicit parent; no fields of its own
  ├─ accounts/{accountId}
  ├─ transactions/{txId}
  ├─ loans/{loanId}
  └─ recurring/{ruleId}
```
`uid` is the FirebaseAuth user id. All three collections are read live as
streams in [`firestore_service.dart`](lib/services/firestore_service.dart)
and held in [`AppState`](lib/state/app_state.dart). Every document is written
with a `createdAt` server timestamp (`FieldValue.serverTimestamp()`) on
create; it is not part of the model `toMap()` and `update*` calls don't touch
it.

### Documents

**accounts/{accountId}** — a money account ([account.dart](lib/models/account.dart))

| field | type | notes |
|---|---|---|
| `name` | string | display name |
| `type` | string | `bank` \| `cash` \| `fd` (fixed deposit) |
| `colorHex` | string | `#RRGGBB`, default `#3DEBA8` |
| `openingBalance` | number | the **only** persisted balance figure; live balance is computed (see "Account balance is computed") |
| `createdAt` | timestamp | server-set; stream ordered by this ascending |

**transactions/{txId}** — income / expense / transfer ([app_transaction.dart](lib/models/app_transaction.dart))

| field | type | notes |
|---|---|---|
| `date` | string | `YYYY-MM-DD` (no time component) |
| `type` | string | `income` \| `expense` \| `transfer` |
| `accountId` | string | source account (the account for income/expense) |
| `toAccountId` | string\|null | transfer destination; null otherwise |
| `category` | string | from [data/categories.dart](lib/data/categories.dart) |
| `note` | string | free text |
| `amount` | number | always positive; sign is derived from `type` |
| `status` | string | `received` \| `pending` (meaningful for income; else `received`) |
| `recurringId` | string\|null | set when auto-generated by a recurring rule (else null) |
| `createdAt` | timestamp | server-set; stream ordered by `date` descending |

**loans/{loanId}** — money lent or borrowed ([loan.dart](lib/models/loan.dart))

| field | type | notes |
|---|---|---|
| `loanType` | string | `lent` \| `borrowed` |
| `who` | string | counterparty name |
| `amount` | number | positive |
| `reason` | string | free text |
| `date` | string | `YYYY-MM-DD` |
| `accountId` | string | account the money moved through |
| `status` | string | `pending` \| `repaid` |
| `createdAt` | timestamp | server-set; stream ordered by this descending |

A pending `lent` loan deducts `amount` from its `accountId` balance until
marked repaid (see "Lent money leaves the account"). Loan repayment flips
`status` to `repaid` via `updateLoan(id, {'status': 'repaid'})` — no separate
transaction document is created.

**recurring/{ruleId}** — fixed monthly income/expense automation ([recurring_rule.dart](lib/models/recurring_rule.dart))

| field | type | notes |
|---|---|---|
| `type` | string | `income` \| `expense` (no transfer) |
| `accountId` | string | account entries are posted to |
| `category` | string | from [data/categories.dart](lib/data/categories.dart) |
| `note` | string | free text |
| `amount` | number | per-month amount |
| `dayOfMonth` | number | 1–31, clamped to month length when generating |
| `status` | string | income only: `received` \| `pending` |
| `active` | bool | paused rules generate nothing |
| `startMonth` | string | `YYYY-MM`, first month to generate |
| `lastGeneratedMonth` | string\|null | high-water mark; null until first run |
| `createdAt` | timestamp | server-set; stream ordered by this descending |

See "Recurring transactions are materialized client-side" below.

### Security rules
[`firestore.rules`](firestore.rules) (rules v2) grants read/write on the whole
`users/{userId}/**` subtree only when `request.auth.uid == userId`. A user can
never see another user's tree. Deploy with:
```
firebase deploy --only firestore:rules --project=fintrack-05220041
```

### Indexes
[`firestore.indexes.json`](firestore.indexes.json) is empty — every stream
sorts on a single field (`createdAt` or `date`), so only Firestore's automatic
single-field indexes are needed. Adding a query that filters + orders on
different fields will require a composite index here.

## Android

- `applicationId = com.example.financialtracker`
- `minSdk = 23` (firebase_auth requirement), `multiDexEnabled = true`
- `ndkVersion = "27.0.12077973"` — pinned because Firebase plugins need 27+
- Debug SHA-1 + SHA-256 are registered with the Firebase Android app; if
  the debug keystore rotates, re-register via the Firebase Management API
  (see `/tmp/fb_setup.py` for the pattern — uses the firebase-tools token
  with `cloud-platform` scope; refresh via the public firebase-tools OAuth
  client creds).

Run on a connected device: `flutter run -d <device-id>`. Get the id from
`flutter devices`. The benign `GoogleApiManager: SecurityException:
Unknown calling package name 'com.google.android.gms'` in logcat is Play
Services chatter, not an app bug — ignore it.

## iOS — deliberately deferred

`firebase_options.dart` keeps PLACEHOLDER iOS values on purpose. When
picking up iOS: run `flutterfire configure --platforms=ios
--project=fintrack-05220041`, register the iOS bundle in Firebase Auth,
and add `GoogleService-Info.plist` reverse client ID for Google Sign-In.

## House rules

- Don't reach for `intl` — the formatters in [utils/formatters.dart](lib/utils/formatters.dart) cover everything.
- Don't introduce a new state-management library. `provider` is enough.
- New persisted fields go on the model + `toMap`/`fromMap`. No JSON codegen.
- Keep `flutter analyze` clean before committing.
- LKR is the only currency; show as `Rs 12,345` via `fmt()` or `fmtFull()`.
- Use `MaterialPageRoute` / `showAppSheet` for new screens/sheets — never push raw widgets that read `AppState` onto the root navigator without re-providing it.
